# ERC-20 Implementation Notes

### Core Idea

An ERC-20 token is a simple state system that updates numbers (balances) based on user actions.

Instead of thinking in complex systems, think like this:


User Action → Validation → State Change → Event Log


This is the entire ERC-20 model.

---

## 1. Core Mental Model (Most Important)

Every operation in ERC-20 follows the same pattern:
```text
Check rules
↓
Update storage
↓
Emit event
```


Nothing else happens.

---

## 2. Balance System (Ledger)

ERC-20 is just a ledger.

```solidity
mapping(address => uint256) public balanceOf;
```
#### Meaning
Address → Token Amount

Tokens are NOT objects.
They are just numbers stored in this mapping.

## 3. Total Supply System

```solidity
uint256 public totalSupply;
```
#### Meaning

Tracks how many tokens exist in the system.
- Mint → increases supply
- Burn → decreases supply

## 4. Transfer Flow 

#### Function

```solidity
transfer(address to, uint256 amount)
```
#### Step-by-step flow
1. Check sender balance
2. Subtract tokens from sender
3. Add tokens to receiver
4. Emit Transfer event

#### Core logic

```solidity
balanceOf[msg.sender] -= amount;
balanceOf[to] += amount;
```

## 5. Allowance System 

Used when someone else spends your tokens.

```solidity
mapping(address => mapping(address => uint256)) public allowance;
```
#### Meaning

```text
Owner → Spender → Allowed Amount
```

#### approve()

##### Purpose
- Give permission to another address.

##### Flow
- Store allowance
- Emit Approval event
```solidity
allowance[msg.sender][spender] = amount;
```

#### transferFrom()
##### Purpose
- Spender moves tokens from another user.

##### Flow
1. Check balance
2. Check allowance
3. Reduce allowance
4. Move tokens
5. Emit Transfer event

#### 6. Minting (Creating Tokens)
##### Flow
1. Increase totalSupply
2. Increase user balance
3. Emit Transfer event (from zero address)

```solidity
balanceOf[to] += amount;
totalSupply += amount;
```

##### Special Rule
```text
address(0) = "no sender"
```
- Used to represent token creation.

#### 7. Burning (Destroying Tokens)
##### Flow
1. Reduce user balance
2. Decrease totalSupply
3. Emit Transfer event (to zero address)

```solidity
balanceOf[from] -= amount;
totalSupply -= amount;
```

#### 8. Events System
```solidity
event Transfer(address indexed from, address indexed to, uint256 value);
event Approval(address indexed owner, address indexed spender, uint256 value);
```
##### Purpose

Events are logs used by:
- wallets
- block explorers
- frontend apps

##### Important Rule
- Events do NOT store state.
- Only the contract storage is real data.

#### 9. Decimals Concept

- Tokens use integers only.
```solidity
token = 10^18 smallest units (if decimals = 18)
```
##### Why?
- Solidity does not support decimals.
- Decimals are only for display.

#### 10. Public Variables Behavior

```solidity
uint256 public totalSupply;
```

##### Automatically creates:
```solidity
function totalSupply() external view returns (uint256)
```

This is just a helper function generated by Solidity

### FINAL MODEL

Every ERC-20 operation follows this simple flow:
```solidity
User Action
    ↓
Validation (checks)
    ↓
State Update (balances, supply, allowance)
    ↓
Event Emission
```
